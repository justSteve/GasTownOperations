Gas Town's operating model is one human coordinating 20-30 autonomous agents on software projects. These agents do not simply run in isolation. They need to discover each other, query each other, report their status, and respond to shared conventions. The question is not "can we run many Claude Code instances?" -- it is "can we make them work together as a coordinated system?" That requires a standard for what it means to be a well-formed agent. That standard is the Zgent.

What Is a Zgent?

A standalone Claude Code agent that has graduated from the Gas Town Academy

Publishes its API — Other Zgents can discover what queries and interactions it supports
Follows shared conventions — Logging, error handling, event streaming, configuration patterns
Independently deployable — Works standalone but integrates seamlessly with the ecosystem
Observable behavior — Hook instrumentation, event emissions, structured logs

A raw Claude Code agent is a brain in a jar.
A Zgent is a complete organism.

These four properties are the graduation criteria from the Gas Town Academy. Any Claude Code agent can become a Zgent by satisfying all four. The key distinction: a raw Claude Code agent can do work, but it cannot participate in a larger system. It has no way to tell other agents what it does. It has no standard way to report its activity. It cannot be discovered, queried, or monitored. A Zgent can do all of these things because it adheres to shared conventions. The metaphor we will develop in this presentation is anatomical: a Zgent is a complete organism with bones, a nervous system, senses, a voice, and a habitat.

The ECC (Everything Claude Code) schema models all 7 artifact types as structured JSON data. Skills give the Zgent capabilities. Hooks wire it to events. The ECC schema is the blueprint language that describes these bones, and the materializer is what produces the actual .claude/ files from that schema.

If artifacts are the skeleton, CrudEngine is the nervous system -- the metabolic process that keeps the skeleton alive. Every time an artifact is created, read, updated, or deleted, CrudEngine manages that operation with full observability. Traffic logging captures every operation with an ID, timestamp, duration, success/failure, and before/after snapshots. Subscriptions allow other systems to react in real time when artifacts change. Versioning means you can always roll back. This is the "neural activity" of a Zgent -- the constant stream of lifecycle operations that keep it configured and current. Without CrudEngine, artifact management is manual and opaque. With it, every change is observed, auditable, and reversible.

The Agent Orchestration Engine provides the sensory apparatus of a Zgent. Structured logging with multiple transports (console, file, remote) gives the Zgent "sight" -- it can see and record what it is doing. The hierarchical error system gives it "pain" -- categorized signals that something is wrong, with appropriate severity levels. The typed event system gives it "hearing" -- the ability to broadcast events and listen for events from other systems. The scene execution framework gives it proprioception -- awareness of where it is in a larger workflow. These four subsystems are the reusable foundation that every Zgent builds on. The AOE is not application-specific; it is the shared sensory infrastructure of the entire ecosystem.

tmuxMOO is the environment where Zgents live. It is built on tmux -- the terminal multiplexer -- but the concept is deeper than terminal management. Each pane is a "room" in the MOO sense: a persistent space where an agent exists, produces output, and can be observed. Multiple Zgents share a session, creating shared presence -- the human and agents cohabit the same text environment. Critically, this is persistent without human presence. You detach from the tmux session and walk away. The Zgents keep working. You reattach later, and the narrative has continued. Agents can observe each other's panes via capture-pane, enabling cross-room communication. The spatial metaphor from MOOs is preserved: you navigate between rooms, each room has inhabitants with behaviors, and the world persists whether you are watching or not.

"A slow dawning event" — The 1995 vision of environments where objects have behavior and respond to input is exactly what modern agent frameworks are rediscovering. The concept was right. The technology finally caught up.
Steve built Virtual Educational Environments on MOO infrastructure in 1995-97. MOO stood for MUD Object-Oriented -- text-based, multi-user environments where everything was a programmable object with "verbs" (behaviors). Rooms persisted. Multiple users shared presence. Natural language was the interface. The vision was right, but the technology -- telnet, limited computing power, no AI -- could not deliver the full promise. Three decades later, every concept maps 1:1 to what we are building. Programmable objects with verbs are agents with defined behaviors. Persistent rooms are tmux panes. Natural language parsing is now powered by LLMs. Multi-user presence is humans and agents sharing workspaces. Telnet was a pipe; tmux is a programmable stage. Steve describes recognizing this as "a slow dawning event" -- the realization that the vision from 1995 has become feasible with 2026 technology. This is not nostalgia. It is architectural continuity across three decades.

A Day in the Life of a Zgent

What happens from startup to collaboration

06:00 — Startup: The Zgent spawns in its tmux pane. CrudEngine initializes its artifacts. AOE logging begins. It announces its presence.
06:01 — API Publication: The Zgent publishes its capabilities. Other Zgents can now discover and query it.
08:30 — Work: An incoming event triggers a hook. The Zgent processes data, emits structured logs, fires events that other Zgents can subscribe to.
10:15 — Collaboration: Another Zgent queries this one via the published API. A structured response is returned. The querying Zgent adapts its behavior based on the answer.
14:00 — Human Interaction: The operator attaches to the tmux session. Reviews the narrative. Issues a command. The Zgent responds in its pane. The operator detaches. The Zgent continues.
This conceptual walkthrough shows a typical day for a well-formed Zgent. At startup, it spawns in its tmux pane and initializes -- CrudEngine loads its artifact configuration, AOE logging starts streaming, and it announces its presence to the ecosystem. It publishes its API so other Zgents know what it can do. During the workday, hooks fire in response to events, the Zgent processes data and emits structured logs. Other Zgents query it through the published API, getting structured responses that inform their own behavior. The human operator can attach and detach from the tmux session at will -- the Zgent does not depend on human presence to continue working. Every operation is logged, every event is emitted, and every interaction is observable. This is the daily rhythm of a well-formed autonomous agent.

The Academy is the process, not a place.
Any Claude Code agent can graduate by meeting the criteria.

The Gas Town Academy is the graduation process. It is not a separate system or a training step -- it is a set of criteria that an agent must satisfy to be considered a well-formed Zgent. A raw Claude Code agent might be capable and functional, but it exists in isolation. It has no standard way to be discovered, queried, or monitored. It handles errors its own way. Its configuration is manual and opaque. A graduated Zgent has adopted all the shared conventions: AOE logging and error handling, CrudEngine-managed artifacts, a published API, and participation in the inter-agent query protocol. The Academy is the quality gate that ensures every agent in the ecosystem can be trusted to behave predictably, report its status, and collaborate with others.
